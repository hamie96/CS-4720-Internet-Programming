Objectives

Be able to implement a simple protocol using sockets.

Be able to describe how services provided by a module map to services provided by a network server. (This is not covered directly but could be a good test question)

Overview

You will make the services of the Controller class you implemented in Assignment 1 available from a network server. You will carry this out by using a simple protocol using sockets as the communication medium.

Detailed Requirements

Protocol

The protocol is as simple as possible.

  • The client connects to the server
  • The client sends a request
  • The server sends a response
  • The connection is terminated

The client request will consist of a single letter followed, in one case, by a numeric string. Here are the requests.

Code Parameter Meaning
A value Add the value to the collector
a   Ask for the average
c   Ask for the count
d   Ask for the standard deviation
q   Ask for the sum of squares
s   Ask for the sum
v   Ask for the variance
z   Reset the collector

The response of the server may consist of a numeric string or a string with the status, depending on the request.

Code response
A “OK” if the value is valid numeric or “Error” if not
a Average as numeric string
c Count as numeric string
d Standard deviation as numeric string
q Sum of squares as numeric string
s Sum as numeric string
v Variance as numeric string
z “Reset”

Important if the request does not begin with a proper character or is empty, then the response should be “Error”.

For example, the client might send the request

A12

to add the value 12 to the collector. The server will carry out that action (presumably) and respond with

Ok

On the other, the client might send the request

Axx12

The server will discover the problem with converting the parameter to a number and respond with

Error

In the case of Error and Ok responses, you may add extra information after the required first word.
This is common with responses: a tag at the beginning indicating ok or error and then the actual response or clarifying information after that.

Implementation

Create a new project and include your Controller class module in the project. You should not make any changes to that class.

The server should begin by creating a controller object. We’ll call it controller for these notes. That name is not required.

Use the Controller class, do not re-implement the functionality in the server.

The ‘z’ command, reset, can be implemented by reinitializing the controller object. So:

controller = Controller()

Testing and Support

It is very helpful to have implementations available as examples and to test against. I am providing a server and three clients written in Java. This illustrates that programs cooperating over a network need not be implemented in the same language.

The programs are packaged as an IntelliJ project (the project will open in Eclipse just fine): assign03-sockets.zip The code includes an implementation of Collector in Java. All of the code is very similar to my version of the assignment in Python.
I tried to avoid obscure coding.

Client1 sends one request and prints the response. This is a good first program to run against your server. If this works, there’s a good chance that the network part is fine.

Clent2 tries out all the requests, including a reset at the beginning.

Client3 is a test program. It will throw an exception if there is an error. If the program detects no problems, it displays a message at the very end saying that. This program is large and messy, so it could be hard to trace back if there is an error, but you should be able to tell from the stack trace what message caused the problem. A version of Client3 will be used as part of the evaluation of your submission.

Evaluation for Grade

Your server will be started and several clients (similar to the Java clients) will be run. Results will be compared to expected. The code will be checked for reasonableness.

Submitting the Assignment

Zip up your project and submit it through the appropriate D2L folder.
